## 已确认事实

### 1. 初始化逻辑

入口函数：

- `init(ContextInfo)`

已确认行为：

- 股票池：`ContextInfo.get_sector('000300.SH')`
- 设置 universe：`ContextInfo.set_universe(ContextInfo.s)`
- 持仓字典：`ContextInfo.holdings = {i: 0 for i in ContextInfo.s}`
- 权重：`[0.1] * 10`
- 手续费率：`0.0003`
- 账户 ID：`testS`

直接含义：

- 股票池是沪深 300 成分股
- 设计目标是最多持有 10 只股票
- 每只股票目标资金占比 10%

### 2. 调仓频率

入口函数：

- `handlebar(ContextInfo)`

关键条件：

```python
if d > 60 and d % 20 == 0:
```

直接含义：

- 前 60 根 bar 主要用于指标窗口准备
- 从第 61 根 bar 之后开始
- 每 20 根 bar 调仓一次

在日线下，这就是“约 20 个交易日一次调仓”。

### 3. 价格与持仓更新

在 `handlebar()` 中：

```python
price = ContextInfo.get_history_data(1, '1d', 'open', 3)
ContextInfo.latest_price = price
```

直接含义：

- 当前代码用于交易和估值的价格字段是 `open`
- 后续买卖、持仓估值、日志输出都基于这组价格

这点很关键，因为它会影响“回测成交价到底按开盘价还是收盘价”的解释。

### 4. 信号函数

入口函数：

- `signal(ContextInfo)`

它读取了三组历史数据：

- `22` 日 `high`
- `2` 日 `high`
- `62` 日 `close`

买入条件：

```python
if data_high_pre[k][-2] > max(data_high[k][:-2]):
    buy[k] = 1
```

卖出条件：

```python
elif data_high_pre[k][-2] < np.mean(data_close60[k][:-2]):
    sell[k] = 1
```

直接含义：

- 买入更像“前一日高点突破过去一段时间高点”
- 卖出更像“前一日高点跌破 60 日均线”

虽然命名叫“多因子”，但从当前代码看，基础买卖触发核心并不是财务因子，而是价格序列条件。

### 5. 排序逻辑

在 `handlebar()` 中，只有 `buys[k] == 1` 的股票才会进入排序：

```python
rank1[k] = ext_data_rank('atr', ...)
rank2[k] = ext_data_rank('adtm', ...)
rank_total[k] = 1.0 * rank1[k]
```

然后：

```python
tmp = sorted(list(rank_total.items()), key=lambda item: item[1])
tmp_stock = [i[0] for i in tmp[:10]]
```

直接含义：

- 排序候选只来自“已经触发买入信号”的股票
- 最终只保留前 10 名
- 当前实际总分只用了 `atr`
- `adtm` 虽然读了，但没有被用进 `rank_total`

这说明：

- 当前策略名称里的“多因子”与实际实现不完全一致
- 更准确地说，它目前更像“价格突破筛选 + 单扩展因子排序”

### 6. 买卖执行逻辑

卖出列表：

```python
sell_list = [k for k in ContextInfo.s if ContextInfo.holdings[k] > 0 and sells[k] == 1]
```

买入列表：

```python
buy_list = [k for k in tmp_stock if ContextInfo.holdings[k] == 0 and buys[k] == 1]
```

卖出行为：

- 用当前价格卖出全部持仓
- 调用 `order_shares(k, -shares, 'fix', sell_price, ...)`
- 更新现金、利润、持仓

买入行为：

- 按当前可用现金重新做资金分配
- `ContextInfo.money_distribution = {k: i * ContextInfo.money ...}`
- 再按分配金额折算成手数
- 调用 `order_shares(k, shares, 'fix', buy_price, ...)`

直接含义：

- 卖出是清仓式卖出
- 买入是等权近似分配
- 买入是在卖出完成后，用剩余现金重新分配

## 可疑 / 待验证点

这些不是“没实现”，而是“需要进一步确认是否符合原设计意图”。

### 1. `adtm` 被读取但没有参与最终排序

现状：

```python
rank2[k] = ext_data_rank('adtm', ...)
rank_total[k] = 1.0 * rank1[k]
```

问题：

- 看起来原意像是多因子组合排序
- 但现在实际上只有 `atr` 生效

这是当前最明显的“命名与实现不完全一致”的地方。

### 2. 使用的是 `open` 而不是 `close`

现状：

```python
price = ContextInfo.get_history_data(1, '1d', 'open', 3)
```

问题：

- 信号函数里参考的是 `high` 和 `close`
- 但执行价格和估值价格来自 `open`

这未必是错，但需要确认是否符合原策略设计。

### 3. `ContextInfo.money_distribution` 按卖出后剩余现金重新分配

这意味着：

- 如果卖出较多，买入资金会扩大
- 如果卖出较少，买入资金会受当前剩余现金限制

这本身合理，但需要确认是否与用户期望一致。

### 4. 是否真的是 QMT 当前实际加载版本

这是目前最重要的非代码问题。

当前还不能直接断言：

- 工作区这份源码
=
- QMT 回测时实际加载执行的那份源码

所以后续任何“为什么回测结果和代码不一致”的判断，都必须先排这个问题。
