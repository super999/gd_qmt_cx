## 二、逐文件代码审查

### 2.1 `python/多因子选股回测示例.py` — 🔴 重度改进

**严重问题（P0）：**

#### P0-1：资金计算存在隐蔽 Bug

```python
# 第 246 行
ContextInfo.money_distribution = {k: i * ContextInfo.money for (k, i) in zip(tmp_stock, ContextInfo.weight)}
```

问题：`ContextInfo.weight = [0.1] * 10`，但 `tmp_stock` 可能不足 10 只。当 `tmp_stock` 只有 5 只时，zip 只取前 5 个权重 0.1，每只分 10% 资金——看起来没问题。但如果 `tmp_stock` 超过 10 只呢？不会，因为前面已经截断了。但**核心问题是**：10 只各分 10%，加起来是 100%，但还有未卖出的旧持仓占着市值。这会导致**资金超分配**。

**建议**：资金分配应基于"本次可用资金"，而非"全部现金"。卖出后的资金回收、手续费消耗都应纳入计算。

#### P0-2：卖出和买入的执行顺序导致资金不准确

```python
# 第 229-244 行：先卖出
for k in sell_list:
    ContextInfo.money += amount - fee
    ...

# 第 246 行：然后用全部 money 做分配
ContextInfo.money_distribution = {k: i * ContextInfo.money ...}

# 第 248-273 行：再买入
for k in buy_list:
    ContextInfo.money -= amount + fee
    ...
```

问题：先卖后买，中间 `ContextInfo.money` 被修改了。但 `money_distribution` 在卖出循环结束后才计算，用的是"已经回收了卖出资金"的 money。这一步本身没问题。但**买入循环中 money 是逐个扣减的**，而 `money_distribution` 是基于扣减前的 money 计算的。这意味着后面买的股票，实际可用资金已经被前面的买入消耗了。

**建议**：改为"先算总需求，再按比例分配"，而非"先分配后逐个扣减"。

#### P0-3：`ext_data_rank` 调用的证券代码拼接可能有问题

```python
rank1[k] = ext_data_rank('atr', k[-2:] + k[0:6], 0, ContextInfo)
```

这里把 `600519.SH` 拼接成 `SH600519`。这种拼接方式是 QMT 扩展数据要求的格式吗？如果 `k` 的格式不是严格的 `XXXXXX.XX`（比如有些带前缀或后缀变体），拼接结果就会错误。

**建议**：添加明确的格式转换函数，而非内联拼接。至少加一行注释说明格式约定。

**中等问题（P1）：**

#### P1-1：全局状态全部挂在 ContextInfo 上，缺乏封装

所有状态变量（`holdings`、`money`、`buypoint`、`trade_log` 等）都直接挂在 `ContextInfo` 上。这是 QMT 框架的限制，但可以在框架限制内做一层封装。

**建议**：创建一个 `Portfolio` 类来管理持仓和资金状态，在 `init` 中初始化，在 `handlebar` 中通过组合方式使用。这样逻辑更清晰，也更容易做单元测试。

#### P1-2：`signal()` 函数缺乏防御性编程

```python
def signal(ContextInfo):
    data_high = ContextInfo.get_history_data(22, '1d', 'high', 3)
    data_high_pre = ContextInfo.get_history_data(2, '1d', 'high', 3)
    data_close60 = ContextInfo.get_history_data(62, '1d', 'close', 3)
    
    for k in ContextInfo.s:
        if k in data_close60:
            if len(data_high_pre[k]) == 2 and len(data_high[k]) == 22 and len(data_close60[k]) == 62:
```

问题：当某只股票数据缺失时，`data_high_pre[k]` 可能长度不为 2。当前用 `if` 跳过了，但**没有记录**哪些股票因为数据不足被跳过了。调试时完全无法得知。

**建议**：对数据不足的股票记录日志，至少在调仓日志中体现"本次有多少股票因数据不足被排除"。

#### P1-3：`rank_total` 只用了 rank1，rank2 浪费了

```python
rank_total[k] = 1.0 * rank1[k]
```

这行乘以 1.0 是无意义的。而且 rank2 被计算出来却完全没有使用。

**建议**：要么真正实现多因子加权，要么删除 rank2 相关代码。当前状态是"半成品"的典型特征——做了但没有做完，也没有标注 TODO。

#### P1-4：重复的 JSON 写入逻辑

`_flush_outputs` 在每次调仓后都被调用，如果回测跨越多根 bar，会反复覆盖同一个文件。这在功能上没问题，但从工程角度看：
- 应该在最后一根 bar 才写最终结果
- 中间调仓可以写增量日志
- 当前实现中 `is_last_bar()` 的异常处理过于宽泛（bare `except Exception: pass`）

**轻微问题（P2）：**

- `_to_float` 对 `None` 的处理：返回 `0.0` 可能掩盖真实的空值问题
- `_safe_date` 对失败的日期返回空字符串，调用方没有检查空值
- `ContextInfo.day` 初始化了但从未使用
- 变量名 `s` 代表股票池，可读性极差

**P1 问题：**

#### P1-1：买入下单在信号判断的同一条 bar 上用开盘价成交

```python
# 第 183-184 行：用昨天收盘价判断信号
prev_close = _to_float(close_list[-2])
ma20 = _to_float(np.mean(close_list[-21:-1]))
ma60 = _to_float(np.mean(close_list[-61:-1]))

# 第 201 行：用今天开盘价成交
order_shares(stock, shares, 'fix', open_price, ContextInfo, ContextInfo.accountID)
```

逻辑上是对的（T 日信号，T+1 开盘成交），但这取决于 QMT `handlebar` 的调用时机。如果 `handlebar` 在收盘时调用，那 open_price 是当天的开盘价，等于是"当天信号当天成交"，存在**未来函数风险**。

**建议**：明确注释 QMT 的 handlebar 调用时机，或用 `is_last_bar` 的前一天信号 + 次日开盘价模式。

#### P1-2：持仓和现金的模拟可能和 QMT 实际不一致

策略自己维护了 `ContextInfo.cash` 和 `ContextInfo.holding`，但 QMT 自己也有账户资金管理。如果两者不同步，会导致：
- 策略认为还有钱，但 QMT 实际已经没钱了
- 策略认为已经卖了，但 QMT 实际没成交

**建议**：至少在关键节点（每次交易后）加一个和 QMT 实际账户的对比检查。

## 三、团队技术提升方案

### 第一层：代码规范与质量底线

#### 1.1 编码规范

当前代码库没有统一的编码规范。建议建立以下基线：

| 规范项 | 当前状态 | 目标状态 |
|--------|----------|----------|
| 文件编码 | 混用 GBK/UTF-8 | 统一 UTF-8（QMT 策略文件可保留 GBK） |
| 变量命名 | `s`、`d`、`k` 等单字母 | 有语义的名称：`stock_pool`、`bar_index`、`stock_code` |
| 函数文档 | 大部分缺失 | 关键函数必须有 docstring |
| 类型提示 | 完全没有 | 核心函数逐步添加 |
| 魔法数字 | `0.95`、`0.0003`、`20`、`60` 散落各处 | 集中到配置区域或常量定义 |

#### 1.2 错误处理原则

当前代码的异常处理几乎都是 `except Exception: pass` 或返回默认值。

**提升目标：**

```python
# 当前写法（不推荐）
def _to_float(value):
    try:
        return float(value)
    except Exception:
        return 0.0  # 0.0 可能是合法值，无法区分"转换失败"和"真正的0"

# 推荐写法
def _to_float(value, default=None):
    """将值转为 float，转换失败返回 default。
    
    Args:
        value: 待转换的值
        default: 转换失败时的返回值，默认为 None（而非 0.0）
    
    Returns:
        float 或 default
    """
    if value is None:
        return default
    try:
        return float(value)
    except (ValueError, TypeError):
        return default
```

#### 1.3 日志规范

当前日志缺乏统一格式。建议建立分级日志规范：

```
[LEVEL] date=YYYY-MM-DD module=xxx action=xxx key=value ...
```

级别定义：
- `CRITICAL`：资金计算异常、下单失败
- `REBALANCE`：调仓触发
- `TRADE`：买卖成交
- `CHECK`：定期状态检查
- `DEBUG`：因子计算细节

### 第二层：架构与设计能力

#### 2.1 策略架构模板

当前策略代码把"数据获取 → 信号计算 → 下单执行 → 状态管理"全部混在一个 `handlebar` 函数里。随着策略复杂度增加，这会变得无法维护。

**建议的分层架构：**

```
Strategy (策略入口)
├── DataLayer (数据层)
│   ├── get_price_data()
│   ├── get_factor_data()
│   └── validate_data()
├── SignalLayer (信号层)
│   ├── compute_signals()
│   ├── rank_stocks()
│   └── generate_orders()
├── ExecutionLayer (执行层)
│   ├── execute_orders()
│   ├── simulate_slippage()
│   └── calculate_fees()
└── RiskLayer (风控层)
    ├── check_position_limit()
    ├── check_drawdown()
    └── check_sector_exposure()
```

#### 2.2 回测与实盘统一接口

当前代码中，回测用的资金计算和 QMT 实际账户是两套逻辑。这会导致回测结果和实盘表现有偏差。

**建议**：设计一个 `Account` 抽象层，回测时用模拟账户，实盘时对接 QMT 真实账户。接口统一，底层实现不同。

#### 2.3 配置与代码分离

当前策略参数（调仓周期、因子权重、手续费率等）散落在代码各处。

**建议**：创建 `config.py` 或 `config.json`：

```python
STRATEGY_CONFIG = {
    "name": "多因子选股",
    "universe": "000300.SH",
    "rebalance_interval": 20,
    "max_holdings": 10,
    "weight_scheme": "equal",
    "fee_rate": 0.0003,
    "factors": [
        {"name": "atr", "weight": 0.6},
        {"name": "adtm", "weight": 0.4},
    ],
    "risk": {
        "max_single_position_ratio": 0.15,
        "max_drawdown": 0.15,
    }
}
```

### 第三层：测试与质量保障

#### 3.1 测试金字塔

当前项目几乎没有自动化测试。建议按以下优先级建立：

```
        ┌─────────┐
        │  E2E    │  ← 回测全流程验证（最低优先级）
        │  Tests  │
        ├─────────┤
        │Integration│  ← API 调用验证（已有 test_xtquant_api_matrix.py）
        │  Tests  │
        ├─────────┤
        │  Unit   │  ← 信号计算、资金计算、因子排名（最高优先级）
        │  Tests  │
        └─────────┘
```

**最高优先级的单元测试：**
1. `signal()` 函数：给定固定数据，验证买卖信号是否正确
2. 资金计算：验证买入后现金减少、卖出后现金增加是否正确
3. 因子排名：验证排序逻辑是否正确
4. `_to_float` / `_safe_date`：边界值测试

#### 3.2 回测验证清单

每次策略修改后，应按以下清单验证：

- [ ] 回测收益率是否与预期一致（不会因为 Bug 虚增收益）
- [ ] 每笔交易的手续费是否正确计算
- [ ] 持仓数量和现金之和是否等于总资产
- [ ] 是否存在"未来函数"（用未来数据做当前决策）
- [ ] 信号触发后是否在下一次合理时机成交

## 六、总结

当前代码库的最大问题不是"功能不对"，而是**缺乏工程化思维**——代码能跑，但不容易看懂、不容易改对、不容易验证改对没有。

团队技术提升的核心不是"学更多 API"或"写更复杂的策略"，而是：

1. **先让现有代码可读、可测试、可验证**
2. **再在此基础上增加复杂度**
3. **每一步都有质量保障**

这个顺序不能反。如果在不稳固的基础上叠加复杂度，最终会变成"谁都不敢改、谁都不知道改完对不对"的状态。

现在正是建立好习惯的最佳时机——代码库还小，改动成本低，收益高。
