# 候选A显式规则草案 v1

## 文档定位

本文件用于把 `n5_r3` 研究阶段得到的 `候选A-严格`：

- `背景分 >= 3`
- `触发分 >= 2`

翻译成一版可执行、可讨论、后续可直接落代码的显式规则草案。

它的作用不是直接取代策略代码，而是作为：

1. 研究结论到策略实现之间的桥梁
2. 下一步写第一版策略代码的输入文档
3. 后续滚动校准时的基线版本

## 当前收口结论

基于当前研究结果：

- 主候选区间：`背景分 >= 3` 且 `触发分 >= 2`
- 对照区间：`背景分 >= 3` 且 `触发分 >= 1`

当前建议先冻结主候选区间，不再继续扩更多评分变体。

## 一、背景层显式规则

背景层的目的，是回答：

- 当前这一天之前，是否已经形成“值得关注的回撤背景”

### 1.1 背景特征与阈值

| feature | feature_cn | 方向 | 阈值 |
| --- | --- | --- | --- |
| `drawdown_from_high_10` | 相对近10日高点回撤 | 更低更像 | `<= -0.044051` |
| `drawdown_from_high_20` | 相对近20日高点回撤 | 更低更像 | `<= -0.050221` |
| `low_vs_ma20` | 最低价相对20日均线偏离 | 更低更像 | `<= -0.024041` |
| `volatility_5` | 近5日收益波动率 | 更高更像 | `>= 0.010574` |
| `ret_5d` | 近5日涨跌幅 | 更低更像 | `<= -0.014103` |

### 1.2 背景打分规则

对以上 5 个条件逐条判断：

- 满足 1 条，记 1 分
- 不满足，记 0 分

得到：

```text
background_score ∈ [0, 5]
```

### 1.3 背景层入围条件

第一版主候选规则：

```text
background_score >= 3
```

### 1.4 背景层当前解释

这条规则的含义不是“已经跌得越多越好”，而是：

- 已经出现一段有质量的回撤
- 但还不是完全失控的大级别下跌定义

当前研究里，`background_score >= 3` 已经足以把很多弱背景样本排除掉。

## 二、触发层显式规则

触发层的目的，是回答：

- 这一天盘中是否真的出现了像样的低点修复结构

### 2.1 触发特征与阈值

| feature | feature_cn | 方向 | 阈值 |
| --- | --- | --- | --- |
| `m5_low_before_last_quarter` | 5分钟最低点是否出现在最后四分之一天之前 | 更高更像 | `>= 1` |
| `m5_low_pos_ratio` | 5分钟最低点出现位置占全天比例 | 更低更像 | `<= 0.319149` |
| `m5_rebound_to_close` | 5分钟最低点到收盘反弹幅度 | 更高更像 | `>= 0.009317` |
| `m5_up_close_streak_after_low` | 5分钟低点后连续抬高收盘的最长根数 | 更高更像 | `>= 4` |
| `m5_up_bar_ratio_after_low` | 5分钟低点后阳线占比 | 更高更像 | `>= 0.5` |
| `m5_close_in_range` | 5分钟收盘在日内振幅中的位置 | 更高更像 | `>= 0.722222` |
| `m1_up_close_streak_after_low` | 1分钟低点后连续抬高收盘的最长根数 | 更高更像 | `>= 5` |

### 2.2 触发打分规则

对以上 7 个条件逐条判断：

- 满足 1 条，记 1 分
- 不满足，记 0 分

得到：

```text
trigger_score ∈ [0, 7]
```

### 2.3 触发层入围条件

第一版主候选规则：

```text
trigger_score >= 2
```

### 2.4 触发层当前解释

这条规则看起来门槛不高，但它是当前研究阶段一个很重要的结论：

- 先把背景筛干净后
- 触发层不需要再设得极端严格
- 只要出现“至少两项像样的修复证据”，就已经足以和弱样本拉开差异

也正因为如此，`候选B` 把触发门槛降到 `1` 后，开始引入边缘样本。

## 三、第一版显式策略候选规则

把背景层和触发层合在一起，得到当前第一版候选规则：

### 3.1 主候选规则

对某个交易日 `t`，如果满足：

```text
background_score(t) >= 3
and
trigger_score(t) >= 2
```

则把 `t` 视为：

```text
第一版低吸反弹候选入场日
```

### 3.2 对照组规则

宽松对照组保留为：

```text
background_score(t) >= 3
and
trigger_score(t) >= 1
```

当前定位：

- 不作为主规则
- 只用于后续比较“触发层是否该放宽”

## 四、当前不建议直接硬编码成硬条件的部分

虽然上面已经是显式规则，但当前仍不建议把所有子特征都进一步改写成“必须满足某几条固定硬条件”。

原因是：

1. 当前样本量还不大
2. `候选A` 命中的日期内部仍有差异
3. 有些日期是强背景 + 中等触发
4. 有些日期是中等背景 + 强触发

所以第一版更合理的实现方式不是：

- 强行挑 2 到 3 个子特征写死为唯一硬条件

而是：

- 保留“两层轻量评分”结构
- 先把区间规则写进代码
- 后面再根据验证结果收缩

## 五、第一版代码实现建议

下一步如果写第一版策略代码，建议采用下面结构：

### 5.1 背景层函数

输入：

- 日线特征

输出：

- `background_score`
- 每个背景子条件是否命中

### 5.2 触发层函数

输入：

- `1m/5m` 盘中特征

输出：

- `trigger_score`
- 每个触发子条件是否命中

### 5.3 主信号函数

输入：

- `background_score`
- `trigger_score`

输出：

- `is_candidate_a`
- `is_candidate_b`

建议写成：

```text
is_candidate_a = background_score >= 3 and trigger_score >= 2
is_candidate_b = background_score >= 3 and trigger_score >= 1
```

## 六、当前闭环点

到这一步，当前主线已经形成完整闭环：

1. 先定义事件标签
2. 再做特征分析
3. 再拆背景分与触发分
4. 再形成候选规则区间
5. 再做候选日期复盘
6. 现在把 `候选A` 翻译成显式规则草案

因此下一步不应该回头继续扩评分模型，而应该进入：

**把本草案翻译成第一版策略代码实现。**

## 七、当前建议冻结的内容

本轮建议先冻结以下事实：

1. 第一版主候选区间：
   - `background_score >= 3`
   - `trigger_score >= 2`

2. 第一版宽松对照组：
   - `background_score >= 3`
   - `trigger_score >= 1`

3. 第一版实现仍保留轻量评分结构，不急着再拆成更死的硬条件

4. 下一轮工作不再扩模型，直接进入第一版策略代码实现
