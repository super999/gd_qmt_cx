**报告标题**  
光大 QMT 学习项目当前阶段分析与下一阶段执行建议

**一、项目背景与核心目标**  
当前工作区的目标，不是从零重写一套量化交易框架，而是沿着 QMT 自带示例策略逐步学习、验证、最小修改，最终写出一个能在 QMT 中成功运行、可解释、可复现的自定义买卖策略。

用户当前更关注的现实目标是：

1.  短期内建立对 QMT 环境、Python 环境、示例策略结构的稳定认知。
2.  尽快形成一个“能跑、能看日志、能解释行为”的最小可用策略版本。
3.  在控制时间投入的前提下，逐步过渡到适合短线操作的策略雏形。
4.  用户希望未来 1 个月内尝试获得“不追求高收益、但尽量稳定”的短线收益。
5.  计划必须明确，同时日常执行不能排得太满，要预留容错空间，避免中断后整体失控。

**二、当前已确认事实**

1.  QMT 安装目录是 D:\\光大证券金阳光QMT实盘。
2.  QMT 示例策略目录是 D:\\光大证券金阳光QMT实盘\\python。
3.  当前工作区是 D:\\codex\_n\_workspace\\gd\_qmt\_cx。
4.  推荐与 QMT 配套使用的 Python 环境是 d:\\python\_envs\\gd\_qmt\_py36，因为它与 QMT 安装目录中的 python36.dll 匹配。
5.  d:\\python\_envs\\gd\_qmt\_env 虽然更适合一般分析，但不应作为 QMT 原生接口首选环境。
6.  D:\\光大证券金阳光QMT实盘\\python\\多因子选股回测示例.py 当前不是可直接阅读的普通源码，更像封装内容。
7.  在没有拿到可读源码、回测日志、成交记录、持仓变化之前，不能对该示例策略做逐函数、逐变量、逐下单调用的精确解释。

**三、当前项目的关键约束**

1.  目前最大的瓶颈不是“策略公式不够高级”，而是输入信息不透明。
2.  示例策略不可读，导致无法准确判断其股票池、调仓频率、买卖条件、仓位控制、风控规则。
3.  如果此时直接让 AI 输出完整策略解读，极易出现臆造接口、误判逻辑、误导后续开发的问题。
4.  因此，当前最重要的工作不是立刻优化收益，而是先把“可观察性”建立起来：源码、日志、成交、持仓、调仓原因，至少要能拿到其中一部分。

**四、对用户短期目标的现实判断**  
用户的真实诉求可以概括为：

“希望在一个月内，基于 QMT 快速形成一个可执行的短线方案，收益不用很高，但尽量稳定，且执行计划不能太重。”

这个目标可以拆成两层：

1.  技术层目标：尽快搭建一个最小可运行、可回测、可解释、可调试的策略执行框架。
2.  交易层目标：不是追求高频、重择时、重预测，而是寻找一种低频、规则清晰、执行负担较小的短线模式。

这里必须指出一个边界：

-   “一个月内稳定赚钱”不能被当成可直接承诺的结果。
-   更现实的目标应当是：“在 1 个月内，建立一个可运行、可复盘、可控风险的短线策略试验流程，并尽量把交易决策从主观拍脑袋，提升为有规则、有记录、有复盘依据的半系统化执行。”

这比直接追求“稳定盈利”更可实现，也更适合作为下一阶段目标。

**五、从项目现状推导出的合理方向**  
基于当前信息，下一阶段不应该走“复杂多因子黑盒优化”，而应该走“低复杂度、强可解释、强日志、低频执行”的路线。

更适合当前阶段的方向是：

1.  先用示例策略或最简替代策略，跑通回测或仿真流程。
2.  优先建立日志输出和行为追踪，而不是一开始追求选股模型复杂度。
3.  先做日级别或接近日级别的短线/波段逻辑，不建议直接做高频日内。
4.  买卖条件尽量少，规则尽量明确，例如：
    -   买入：满足少数几个明确条件
    -   卖出：止损、止盈、超时退出三种硬规则
5.  控制交易频率，避免因为频繁下单、频繁调参而让执行成本和认知成本失控。

**六、建议的下一阶段目标定义**  
建议把下一阶段定义为：

“用最小工作量，建立一个可跑通、可记录、可复盘的短线策略实验框架，并形成 1 个月内可执行的轻量计划。”

这比“一个月稳定赚钱”更适合作为阶段目标，因为它既服务于赚钱目标，又避免一上来就陷入不切实际的收益承诺。

**七、建议的下一阶段任务顺序**  
**阶段 A：先恢复输入**  
目标：解决“看不清”的问题。

优先任务：

1.  确认 QMT 是如何加载 多因子选股回测示例.py 的。
2.  尝试获取该示例策略的可读源码，或者找到同类未封装示例。
3.  如果拿不到源码，则优先收集：
    -   回测日志
    -   成交记录
    -   持仓变化
    -   调仓日期与结果
4.  基于运行结果反推策略行为，而不是空想代码逻辑。

**阶段 B：建立最小可解释策略**  
目标：先有一个“能解释自己在干什么”的策略。

优先任务：

1.  沿用原流程、原函数名、原变量，做最小改动。
2.  增加结构化日志，至少输出：
    -   当前日期
    -   当前持仓
    -   候选股票池
    -   买入原因
    -   卖出原因
    -   下单结果
3.  补齐最基础的卖出逻辑与资金分配逻辑。
4.  如果买入逻辑不清楚，先做最简单替代版本，也要保证规则明确。

**阶段 C：转成“轻量短线试验版”**  
目标：让它具备短线试验价值。

建议采用的风格：

1.  低频短线，不追求日内超高频。
2.  单次持有周期可先控制在数天到两周以内。
3.  必须有止损。
4.  必须有超时退出。
5.  持仓数量不要太多，避免难以跟踪。
6.  每次只改少量规则，避免调参失真。

**八、面向用户目标的执行原则**  
如果用户的核心诉求是“短期做一些短线，尽量稳定，时间又不能太满”，那么策略研发与执行必须遵守以下原则：

1.  低频优先，少看盘，减少精力消耗。
2.  规则优先，减少临盘主观决策。
3.  风控优先，先控制回撤，再谈收益。
4.  可复盘优先，每次交易都要能回答“为什么买、为什么卖”。
5.  计划留白，不要把每天安排得太满，否则一旦中断就失去连续性。
6.  先求稳定执行，再求收益优化。

**九、建议的 1 个月轻量计划框架**  
这是一个“不过满”的框架，重点是可完成。

**第 1 周：环境与输入确认**

1.  确认 QMT 使用的 Python 环境是否稳定指向 gd\_qmt\_py36。
2.  确认示例策略能否正常加载与运行。
3.  尽量拿到可读源码；若拿不到，就保存回测日志、成交、持仓结果。
4.  目标产出：一份“示例策略实际行为记录”。

**第 2 周：做最小可解释修改**

1.  在不破坏原结构前提下，增加日志。
2.  把买卖触发、仓位变化、调仓原因打出来。
3.  做一次最小回测或仿真。
4.  目标产出：一版“看得懂自己在干什么”的策略。

**第 3 周：改成轻量短线试验逻辑**

1.  简化选股条件。
2.  增加明确的止损、止盈、超时退出。
3.  控制持仓数与交易频率。
4.  目标产出：一版“可用于短线试验”的规则化策略。

**第 4 周：只做复盘，不做大改**

1.  观察交易次数、胜率、单笔盈亏、最大回撤。
2.  找出最影响结果的 1 到 2 个问题。
3.  只允许小修，不允许大改框架。
4.  目标产出：下一轮优化清单，而不是继续无节制调参。

**十、对“稳定盈利”目标的修正建议**  
建议把目标从：

“一个月内尽量稳定赚钱”

修正为：

“一个月内建立一套低频、规则清晰、可执行、可复盘的短线试验系统，并尽量把风险控制在可接受范围内。”

原因是：

1.  这更符合当前项目成熟度。
2.  这能减少因为追求短期收益而导致的过度交易。
3.  这更适合时间不能排太满的现实约束。
4.  这能给后续真正优化收益打基础。

**十一、结论**  
当前阶段最重要的，不是让 AI 直接设计一个复杂盈利策略，而是先把 QMT 示例策略和运行结果变成“可观察、可解释、可修改”的对象。

对用户而言，最合理的下一阶段不是“盲目追求一个月稳定赚钱”，而是：

1.  恢复输入信息。
2.  建立最小可解释策略。
3.  做低频短线试验框架。
4.  用轻量节奏执行与复盘。

如果这个顺序做对了，后面才有可能在较低时间压力下，逐步接近“稳一些”的短线目标。
