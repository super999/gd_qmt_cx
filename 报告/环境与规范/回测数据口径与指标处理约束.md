## 一、复权处理的当前结论

### 1. 外部 Python 回测里，不要假设会自动复权

在 QMT 图形界面或某些内置策略环境里，复权常常是界面或主图上下文的一部分。  
但在你用 `xtquant` / `xtdata` 自己写外部 Python 回测时，应该采用更保守的工程假设：

- 复权必须在取行情数据时显式指定
- 不能默认认为系统会自动帮你统一处理

### 2. 当前可用的复权控制方式

官方 `xtdata` 文档中，至少这些接口支持 `dividend_type` 参数：

- `xtdata.get_market_data`
- `xtdata.get_local_data`
- `xtdata.get_market_data_ex`
- `xtdata.call_formula`
- `xtdata.call_formula_batch`

常见可选值：

- `none`：不复权
- `front`：前复权
- `back`：后复权
- `front_ratio`：等比前复权
- `back_ratio`：等比后复权

说明：

- 不同接口文档里对复权选项的列法略有差异
- 在当前项目里，回测代码应先统一使用同一种复权口径，不要混用

### 3. 第一版回测的默认复权口径

当前建议：

- 日线股票回测，默认使用 `dividend_type='front'`

原因：

- 更适合连续价格序列分析
- 更适合均线、RSI、突破、涨跌幅等价格型技术指标
- 更接近你在 QMT 图上看连续走势时的直觉

### 4. 当前禁止的做法

后续 agent 写回测代码时，禁止出现下面这些情况：

- 取行情时没有明确写 `dividend_type`
- 一部分数据用前复权，一部分数据用不复权
- 买卖逻辑用一套价格口径，指标计算用另一套价格口径
- 复权方式写死在隐蔽位置，导致后续难以核查

### 5. 当前推荐的工程写法

回测代码里应显式有一个统一参数，例如：

```python
PRICE_ADJUSTMENT = "front"
```

然后所有历史行情读取统一使用：

```python
xtdata.get_local_data(..., dividend_type=PRICE_ADJUSTMENT)
xtdata.get_market_data_ex(..., dividend_type=PRICE_ADJUSTMENT)
```

### 6. 如果需要核查复权是否合理

官方提供：

- `xtdata.get_divid_factors(stock_code, start_time='', end_time='')`

当前项目对它的结论是：

- 接口本身可调用
- 当前测试标的上返回空 `DataFrame`
- 这不等于接口不可用，只说明当前样本没有拿到除权数据

因此它的定位应当是：

- 校验工具
- 复权抽查工具

而不是第一版回测的硬依赖。

## 二、指标处理的当前结论

### 1. 不要假设 xtdata 自带通用 Python 指标库

当前没有证据表明你可以像下面这样直接写：

```python
xtdata.MA(...)
xtdata.RSI(...)
```

因此后续 agent 不应臆造这类接口。

### 2. 技术指标的首选方案：自己算

对于第一版外部 Python 回测，技术指标建议全部基于行情数据自己计算。

优先基于这些接口：

- `xtdata.get_local_data`
- `xtdata.get_market_data_ex`

这些接口当前已经实测可用，并且返回结构适合进入 `pandas` 计算流程。

当前优先建议自己实现的指标：

- `MA`
- `EMA`
- `RSI`
- 涨跌幅
- N 日新高 / 新低
- 成交量均线
- 持有天数

### 3. 为什么第一版要自己算

原因很实际：

- 最透明
- 最容易调试
- 最容易和日志对应
- 不依赖 QMT 公式环境差异
- 不会因为公式模型、参数传递或投研端限制产生额外不确定性

### 4. 成交量类指标怎么处理

成交量相关逻辑当前直接从行情字段取：

- `volume`
- `amount`

可用于：

- 量比类近似逻辑
- 成交量均线
- 放量 / 缩量判断
- 金额过滤

当前结论：

- 这类指标不需要额外依赖财务数据
- 也不需要额外依赖公式模型

### 5. QMT 公式指标能不能直接复用

官方确实提供了调用模型的接口：

- `xtdata.call_formula`
- `xtdata.call_formula_batch`

并且文档示例中直接提到了 `MA` 这类模型参数写法。

但当前项目对这条路线的判断是：

- 它更适合“已有公式模型并希望复用”
- 它不应作为第一版股票回测的默认路径

第一版不优先使用它的原因：

- 你当前目标是先把最小回测逻辑跑通
- 自己算指标更容易排错
- 公式模型链路会引入额外环境依赖

### 6. 对后续 agent 的明确约束

如果用户没有明确要求“必须复用 QMT 内置公式指标”，则默认：

- 不走 `call_formula`
- 不走 `call_formula_batch`
- 优先自己用 `pandas` / `numpy` 计算指标

## 三、财务指标的当前结论

### 1. 财务指标不是第一版回测的基础设施

当前项目已经实测：

- `xtdata.download_financial_data` 会阻塞

因此后续 agent 必须遵守：

- 第一版回测不能把财务数据作为前置依赖
- 第一版股票策略不能默认依赖 PE、PB、ROE、净利润增速、营收增速等财务指标

### 2. 为什么财务指标要暂缓

不是因为它不重要，而是因为当前有两个现实问题：

1. 下载接口本身阻塞
2. 财务数据天然涉及披露日、报告期、避免未来函数等复杂问题

在这两个问题没单独打通前，直接把财务指标塞进回测，会让结果很不可信。

### 3. 市盈率 / 财务因子的当前策略

当前默认策略：

- 第一版回测完全排除财务指标
- 如果后续确实要引入财务因子，必须先单独做“财务数据链路验证”

这一步至少要单独确认：

- 财务下载是否稳定
- 财务表结构是否明确
- 使用的是报告期数据还是披露期可见数据
- 回测里是否避免了未来函数

## 四、当前推荐的回测数据处理路线

如果接下来要写股票回测代码，当前推荐路线是：

### 路线 A：第一版默认路线

1. 用 `download_history_data` 先下载历史行情
2. 用 `get_local_data(..., dividend_type='front')` 读取历史行情
3. 用 `pandas` 自己算 MA / RSI / 涨跌幅 / 成交量均线
4. 用 `get_instrument_detail` 做基础属性过滤
5. 用 `get_sector_list` / `get_stock_list_in_sector` 做股票池
6. 输出完整交易日志和指标日志

### 路线 B：后续增强路线

当且仅当第一版稳定后，再考虑：

1. 增加 `call_formula` 复用公式模型
2. 增加更复杂的盘中订阅指标
3. 单独打通财务数据链路

## 五、写回测代码时必须显式落地的配置

后续 agent 写回测代码时，至少必须把这些参数显式写在配置区：

```python
PRICE_ADJUSTMENT = "front"
DATA_SOURCE = "local"
USE_FINANCIAL_DATA = False
INDICATOR_MODE = "python"
```

这些参数的含义：

- `PRICE_ADJUSTMENT = "front"`
  表示统一使用前复权
- `DATA_SOURCE = "local"`
  表示历史回测优先用本地数据
- `USE_FINANCIAL_DATA = False`
  表示第一版不依赖财务指标
- `INDICATOR_MODE = "python"`
  表示第一版指标由 Python 自行计算

3. 默认使用前复权历史数据
4. 默认自己算技术指标
5. 默认不接财务数据
6. 如果要偏离这些规则，必须在代码或文档里写清楚原因

## 七、当前最重要的结论摘要

- 外部 Python 回测里，复权应显式处理，不应假设自动复权
- 第一版股票回测默认统一使用前复权
- 技术指标第一版全部自己算
- 成交量指标直接基于 `volume` / `amount`
- 财务指标第一版先不要接
- `get_divid_factors` 可以作为复权抽查工具，但不是首版主依赖
- `call_formula` / `call_formula_batch` 暂不作为首版默认方案
