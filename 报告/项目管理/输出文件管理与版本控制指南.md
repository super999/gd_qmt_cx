## 4. 版本控制建议

### 4.1 当前 .gitignore 分析

**当前忽略规则**：
```
# IDE
.vscode/
.idea/

# Python
__pycache__/
*.pyc
*.pyo
*.pyd

# Environment
.env
venv/
env/
```

**问题**：
- ❌ **没有忽略 `outputs/` 目录**
- ❌ **没有忽略 `*.json` 输出文件**
- ❌ **没有忽略 `*.txt` 报告文件**

### 4.2 建议的 .gitignore 更新

**方案 A：忽略整个 outputs 目录（推荐）**

```gitignore
# 输出文件目录
code/run_xtquant/outputs/
python/outputs/

# 或者更通用的
**/outputs/
```

**方案 B：忽略特定类型的输出文件**

```gitignore
# 回测输出文件
**/backtest_*.json
**/backtest_*.txt

# 测试输出文件
**/smoke_test_*.json

# QMT 输出文件
**/qmt_*.json
```

### 4.3 建议的版本控制策略

| 类型 | 是否提交 | 说明 |
|------|----------|------|
| 源代码（*.py） | ✅ 是 | 核心资产 |
| 配置文件（*.md, *.json） | ✅ 是 | 文档和配置 |
| **回测输出（*.json）** | ⚠️ 可选 | 建议忽略，文件大且可重建 |
| **报告文件（*.txt）** | ⚠️ 可选 | 建议忽略，可从 JSON 生成 |
| 测试输出 | ❌ 否 | 临时文件，无长期价值 |

### 4.4 特殊情况：需要提交的输出文件

**如果需要保留某些关键回测结果作为基线**：

建议创建单独的目录：
```
baseline/                # 基线结果（提交到 git）
├── 20260422_510300_v1/  # 版本化的基线
│   ├── metrics.json
│   ├── trades.json
│   └── report.md
└── README.md             # 基线说明
```

**基线文件应包含**：
1. 回测参数
2. 关键指标
3. 简要说明
4. 对应的代码版本（Git commit hash）

## 5. 需要审计和修正的文件

### 5.1 已发现的问题

#### ⚠️ 问题 1：.gitignore 不完整

**当前状态**：
- 没有忽略 `outputs/` 目录
- 没有忽略 `*.json` 输出文件

**风险**：
- 输出文件可能被意外提交
- 仓库体积膨胀
- 难以区分代码和输出

**建议修正**：
更新 `.gitignore`，添加：
```
# 输出文件目录
**/outputs/

# 或者更细粒度的控制
# **/backtest_*.json
# **/smoke_test_*.json
```

---

#### ⚠️ 问题 2：outputs 目录无 .gitkeep

**当前状态**：
- `outputs/` 目录可能不存在
- 如果目录被清理，下次运行可能报错

**建议修正**：
在每个 `outputs/` 目录添加 `.gitkeep` 文件（空文件），确保目录结构保留：

```
code/run_xtquant/outputs/.gitkeep
python/outputs/.gitkeep
```

---

#### ⚠️ 问题 3：回测脚本输出路径硬编码

**当前状态**：
- 输出路径在脚本中硬编码
- 不同脚本输出到同一目录

**建议**：
- 保持现状（当前设计合理）
- 但确保 `.gitignore` 正确配置

---

### 5.2 建议的修正操作清单

| 操作 | 文件 | 优先级 |
|------|------|--------|
| 更新 .gitignore | `.gitignore` | 🔴 高 |
| 添加 .gitkeep | `code/run_xtquant/outputs/.gitkeep` | 🟡 中 |
| 添加 .gitignore | `python/outputs/.gitignore`（如果目录不存在） | 🟡 中 |
| 清理旧文件 | `smoke_test_report.json` | 🟢 低 |

## 6. 审计和回溯指南

### 6.1 如何追溯某次回测

**步骤**：

1. **确定时间戳**
   - 从文件名提取时间戳：`backtest_metrics_20260422_144312.json` → `20260422_144312`

2. **查找关联文件**
   - 所有同一时间戳的文件属于同一次回测：
     - `backtest_metrics_*` → 指标
     - `backtest_trades_*` → 交易记录
     - `backtest_equity_*` → 权益曲线
     - `backtest_report_*` → 分析报告

3. **查看报告内容**
   - `backtest_report_*.txt` 包含格式化的分析报告

4. **对比不同回测**
   - 使用时间戳区分不同回测
   - 对比 `metrics.json` 中的关键指标

### 6.2 审计清单

每次提交代码前，检查：

- [ ] 源代码是否已提交
- [ ] 输出文件是否已忽略（或确认要提交）
- [ ] 新生成的输出文件是否有时间戳
- [ ] `.gitignore` 是否完整

### 6.3 回溯示例

**场景**：想知道 2026-04-22 下午 2:43 的回测结果

**步骤**：
1. 查找时间戳：`20260422_144312`
2. 查看文件：
   - `backtest_metrics_20260422_144312.json` → 关键指标
   - `backtest_trades_20260422_144312.json` → 每笔交易
   - `backtest_report_20260422_144312.txt` → 详细报告

## 7. 总结

### 7.1 关键发现

1. **两次回测结果相同**：
   - 不复权 vs 前复权结果完全相同
   - 原因：回测期间 510300 可能没有分红
   - **但复权问题在更长时间区间仍然重要**

2. **版本控制问题**：
   - `.gitignore` 不完整，没有忽略输出文件
   - 输出文件可能被意外提交

3. **文件管理问题**：
   - 有一个无时间戳的旧文件：`smoke_test_report.json`
   - 权益曲线文件可重建，但占用空间

### 7.2 建议的行动项

| 优先级 | 行动 | 预期效果 |
|--------|------|----------|
| 🔴 高 | 更新 `.gitignore`，忽略 `outputs/` 目录 | 防止输出文件被意外提交 |
| 🟡 中 | 在 `outputs/` 目录添加 `.gitkeep` | 确保目录结构保留 |
| 🟢 低 | 删除无时间戳的旧文件 | 减少混乱 |
| 🟢 低 | 可选：删除权益曲线文件 | 节省空间（可从交易记录重建） |

### 7.3 下一步建议

1. **立即**：更新 `.gitignore`
2. **提交前**：检查 `git status`，确保只提交代码
3. **长期**：考虑建立基线回测机制，保留关键结果
