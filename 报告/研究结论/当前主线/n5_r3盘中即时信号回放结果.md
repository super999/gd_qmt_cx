# n5_r3 盘中即时信号回放结果

## 文档定位

本文件记录一次关键语义修正后的验证。

用户已确认：盘中策略必须是“两层门控”：

1. 第一层：日线背景门控
2. 第二层：盘中 V 型触发

并且日线门控采用：

```text
昨日以前的已完成日线 + 当前盘中价格估算当日日线背景
```

也就是说，本策略不是“只看盘中”，而是：

```text
日线背景先成立或接近成立
然后盘中 V 型反转才触发预警
```

本轮不是继续扩模型，也不是继续调参，而是验证：

**当前 `候选A/候选B` 的日线背景 + 盘中触发规则，能否直接迁移成盘中即时买入信号。**

## 回放规则

- 脚本：`code/backtest/backtest_intraday_v_reversal_signal.py`
- 标的：`510300.SH`
- 区间：`20240101` 至 `20260424`
- 数据：`1d` + `1m` + `5m`
- 复权：前复权 `front`
- 日线门控模式：`intraday_estimated_daily_gate`
- 信号计算：
  - 每根已完成 `5m` K 线后推进一次
  - 只使用当前时刻已完成的 `1m/5m` K 线
  - 日线背景使用昨日以前的日线 + 当前盘中价格估算
  - 当日收盘相关字段使用当前 5m 收盘价作为第一版预估收盘价
- 信号门控：
  - 先计算 `background_score`
  - 再计算 `trigger_score`
  - 候选A要求 `background_score >= 3` 且 `trigger_score >= 2`
  - 候选B要求 `background_score >= 3` 且 `trigger_score >= 1`
- 买入模拟：
  - 盘中第一次满足候选规则时发出信号
  - 下一根 `5m` K 线开盘价模拟买入
- 卖出模拟：
  - `3%` 止盈
  - `3%` 止损
  - 跌破信号低点 `0.1%` 容忍线
  - 最多持有 `5` 个交易日

## 关键观察

1. 盘中即时版仍然看日线，且日线背景分是第一层门控。
2. 即便有日线门控，当前即时版表现仍明显弱于收盘确认版。
3. 候选A盘中首次触发 `20` 次，实际不重叠交易 `14` 笔，胜率只有 `28.57%`。
4. 候选B盘中首次触发 `22` 次，实际不重叠交易 `14` 笔，胜率同样只有 `28.57%`。
5. 很多信号在早盘快速触发，但当时 V 型修复并不充分，后续容易触发信号低点破位。
6. 这说明原先基于完整日内结构得到的阈值，不能直接迁移成“盘中一出现就买”的规则。

## 闭环结论

当前结论必须区分两件事：

1. `候选A-严格 + n5_r3退出` 作为收盘确认 / 次日开盘策略，仍是当前回测主基线。
2. 采用“盘中估算日线门控 + 盘中V型触发”后，它仍不能直接作为即时买入策略投入模拟盘或实盘。

因此，当前不能建议：

- 盘中一触发候选A就手工买入
- 直接写自动买入程序
- 把候选B升为主规则

当前可以建议：

- 写盘中监控程序，但第一版只做观察和日志
- 信号分为“盘中预警”而不是“立即买入指令”
- 日志必须同时输出日线背景分和盘中触发分
- 继续记录信号触发后是否真正形成收盘确认

## 下一步建议

为了尽快闭环，不继续无限研究，下一步只做一个工程闭环：

1. 写盘中监控程序，输出候选A/B盘中预警。
2. 预警只记录日志，不直接作为买入指令。
3. 每次预警必须记录：
   - 日线背景分
   - 日线背景命中项
   - 盘中触发分
   - 盘中触发命中项
4. 盘后对比：
   - 盘中首次预警时间
   - 收盘是否仍满足候选A
   - 次日是否继续有效
5. 累积一小段真实盘中日志后，再决定是否把“盘中预警”升级为“盘中买入信号”。
