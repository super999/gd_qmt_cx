# 回测策略开发约束

## 目的

本文件用于给后续 agent / 新会话提供统一的开发前提，避免重复摸底 MiniQMT 环境、`xtdata` API 可用性和当前已知限制。

本文件不替代官方文档，也不替代 API 实测台账。  
使用方式：

1. 先看本文件，理解当前开发边界
2. 再看 `报告/MiniQMT_API可用性清单.md`，确认具体 API 的可用性与返回值格式
3. 最后再回到官方文档核对参数细节

## 当前明确不要依赖的接口

### 1. 财务数据接口

- `xtdata.download_financial_data`

当前状态：

- 本地已复现阻塞
- 在 25 秒超时保护内没有返回

开发约束：

- 第一版策略不要依赖财务因子
- 不要把财务下载作为回测初始化的一部分

### 2. 交易日历接口

- `xtdata.get_trading_calendar`

当前状态：

- 当前客户端直接报错
- 错误信息为 `function not realize`

开发约束：

- 不要把它作为当前版本的交易日历来源
- 若回测逻辑需要交易日判断，应先采用替代方案

### 3. 周期列表探测接口

- `xtdata.get_period_list`

当前状态：

- 当前客户端直接报错
- 错误信息为 `function not realize`

开发约束：

- 不要依赖该接口动态探测周期支持情况
- 周期先按官方文档和已实测场景手工维护

## 当前推荐的数据来源方案

如果目标是写“股票回测代码”，推荐按下面顺序使用数据接口：

### 1. 首选：`xtdata.get_local_data`

适用场景：

- 回测
- 批量历史行情读取
- 需要避免实时订阅依赖的场景

原因：

- 已实测可用
- 返回结构对回测友好
- 更符合“先下载、后本地取数”的稳定模式

### 2. 次选：`xtdata.get_market_data_ex`

适用场景：

- 单标的或小股票池的历史行情读取
- 调试策略逻辑
- 盘中订阅后继续读取拼接后的行情

原因：

- 已实测可用
- 返回 `dict[股票代码 -> DataFrame]`
- 相对容易直接喂给策略计算逻辑

### 3. 补充：`xtdata.get_market_data`

适用场景：

- 做底层验证
- 对比不同取数接口返回结构

原因：

- 已实测可用
- 但第一版回测代码通常不必优先选它

## 当前推荐的策略范围

第一版策略建议控制在“纯行情驱动”的范围内，不要过早扩展。

建议范围：

- 日线级别
- 股票或板块成分股范围
- 基于价格、涨跌幅、均线、简单突破、持有期等规则
- 允许使用板块接口构建股票池
- 允许使用 `get_instrument_detail` 做基础过滤

不建议第一版就做：

- 财务因子策略
- 高频盘中复杂订阅策略
- 多数据源混合策略
- 强依赖交易日历精确推导的复杂调仓逻辑
- 依赖 ETF / 可转债 / 新股专项接口的策略

## 当前推荐的回测代码结构

第一版回测代码建议采用最小可运行结构：

1. 参数区
   - 股票池来源
   - 回测开始/结束日期
   - 周期
   - 调仓频率
   - 仓位数

2. 数据准备区
   - 先调用 `download_history_data`
   - 再调用 `get_local_data` 或 `get_market_data_ex`
   - 明确记录每个标的是否成功取到数据

3. 信号计算区
   - 单独封装选股逻辑
   - 单独封装买入条件
   - 单独封装卖出条件

4. 回测主循环
   - 按日期迭代
   - 先处理卖出
   - 再处理买入
   - 记录现金、持仓、净值变化

5. 输出区
   - 交易记录
   - 每日持仓
   - 每日净值
   - 调试日志

## 当前代码风格约束

- 优先最小实现，不要一开始设计过度复杂的框架
- 优先可解释，不要引入黑盒计算
- 优先强日志，每一步要能看出为什么买、为什么卖、为什么跳过
- 优先结构清晰，把“数据获取 / 信号生成 / 回测执行 / 输出结果”分开
- 不要臆造未验证的 QMT / xtdata API

## 当前交易日历替代方案

由于 `xtdata.get_trading_calendar` 当前不可用，短期内如果回测逻辑需要交易日判断，可采用以下替代方案之一：

1. 直接以行情数据中实际存在的日期索引作为交易日序列
   - 这是当前最推荐的方案
   - 对日线回测尤其简单可靠

2. 手工维护交易日列表
   - 仅适合非常小范围测试

3. 引入外部交易日历
   - 只有当你后续确实需要更严格的交易日控制时再考虑

## 当前最适合交给后续 Agent 的任务

如果下一个 agent 要继续工作，最适合直接接手的任务是：
