- `D:\python_envs\gd_qmt_env`

  - 可读源码
  - 或回测日志、成交记录、持仓变化结果

## 4. 后续工作原则

- 优先基于事实，而不是基于猜测
- 优先最小改动，不轻易重构成另一套架构
- 尽量保持原函数名、原变量、原调用风格
- 不编造 QMT 不存在的 API
- 在不确定接口名时，优先沿用原文件已有调用方式
- 若只有运行结果没有源码，可以先通过日志和成交结果反推策略行为

## 5. 分阶段路线图

### 第 1 阶段：拿到可读源码或回测日志

优先级最高。后续无论是解释策略，还是补买卖逻辑，都需要一个可信输入。

优先提供：

1. `多因子选股回测示例.py` 的可读源码
2. 如果没有源码，提供回测日志
3. 如果日志也不完整，提供成交记录、持仓变化、资金变化截图

### 第 2 阶段：解释示例策略实际在做什么

一旦拿到源码或足够的运行结果，就要明确：

- 调仓频率怎么控制
- 股票池如何生成
- 因子如何打分和排序
- 资金如何分配
- 卖出逻辑怎样触发
- 买入逻辑是否完整

### 第 3 阶段：补齐买卖逻辑与调试日志

在不大改架构的前提下：

- 如果买入逻辑缺失，则补齐真实下单逻辑
- 增加标准化日志，便于看出哪天买、哪天卖
- 让调仓前后持仓、现金、候选池、最终买卖结果都能直接从日志中读懂

### 第 4 阶段：跑通一个自己的 QMT 买卖策略

目标不是只改示例，而是最终完成：

- 有自己的选股条件
- 有自己的买卖条件
- 有基本风控
- 有可解释日志
- 能在 QMT 中稳定回测并验证行为

## 6. 运行后应该如何解释策略行为

后续一旦你跑了策略，解释时至少要回答这几个问题：

- 哪天发生了调仓
- 调仓前持仓里有什么
- 本次候选股票池有哪些
- 最终决定买哪些、卖哪些
- 每笔买卖的价格、数量、金额是多少
- 调仓后还剩多少现金
- 调仓后持仓变成了什么

如果这些信息在策略日志里看不出来，就说明策略的“可解释性”还不够，后续应该优先补日志，而不是急着增加更复杂的因子。

## 7. 当前最需要的输入

为了进入下一步的精确分析或改造，优先提供以下任一内容：

1. `多因子选股回测示例.py` 的可读源码
2. 该策略实际跑出的完整回测日志
3. 回测成交记录、持仓变化、资金变化结果

在拿到这些输入之前，所有针对该策略内部实现的说明都只能采用保守模式，不能假装已经看到了真实源码。
